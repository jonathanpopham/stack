agentic engineering · systems · evals

I ship production software by running fleets of AI agents under a deterministic engineering discipline.

Not prompt-and-pray. A plan becomes a dependency-graphed task system, PageRank-style triage picks the critical path, agent swarms execute in parallel worktrees, and nothing merges until it clears a verification gate and an adversarial audit. The output is boring in the best way: reproducible, tested, and honest about what it can't prove.

The methods, and why each exists

the discipline that makes agent fleets trustworthy at senior scale

Plan-space, not code-space

idea → plan → beads

Rewriting a plan is cheap; rewriting a shipped system is not. Ideas are elaborated into a granular, dependency-linked bead graph — each task carrying its own rationale, so an agent (or a future me) never has to reconstruct intent.

Result: a big swarm can be pointed at the graph and come out close to right, because the thinking already happened.

PageRank on the work itself

graph triage

Tasks form a dependency graph, so I run graph analysis on the backlog — bottleneck detection, cycle checks, and next-action ranking — instead of guessing what to do next. bv --robot-plan emits parallel execution tracks; the swarm fans out along them.

Eval-first, golden-regression

correctness as data

The harness is built before the feature. Behavior is pinned with golden snapshots and per-case baselines, so a regression is a diff, not a surprise. When a number moves, I know exactly which case and why.

The one-script gate

verify.sh · CI is a thin wrapper

A single portable script is the gate — format, lint (warnings are errors), tests, evals, and a real dogfood run of the product on a real workload. CI does nothing the script doesn't, so it's reproducible on a laptop and provable offline with act.

Multi-model adversarial swarms

orchestration

Different model architectures have different blind spots. I run them against each other — generate, cross-score, reveal, rebut — and treat disagreement as signal. Coordination via isolated worktrees and a shared task graph, not a shared branch.

Adversarial audit as a step

distrust the trust core

The most trust-critical component gets an agent whose only job is to forge, confuse, or break it. On my last build the audit found a real authenticity gap in a "tamper-evident" format — so I published the finding in the project's own README. Owning your worst finding reads as competence, not weakness.

Zeugnis

public · OSS

An execution-based verifier for code migration: runs a legacy program and its replacement over the same inputs, compares them at the observable boundary under a declared assumption ledger, and issues a deterministic, hash-chained equivalence certificate — including what it could not prove.

Rust · std-only, zero deps · #![forbid(unsafe_code)]

Who's behind it

I'm a systems and AI engineer who was the founding engineer at a code-intelligence startup — hired pre-product, helped build the engineering team, and designed the parsing, classification, LLM-routing, and evaluation stack that took it to production.

Since then I've gone deep on agentic engineering: treating AI coding agents as a fleet to be orchestrated with real discipline — dependency-graphed planning, eval-first correctness, verification gates, and adversarial review — rather than a chat box. The public repos above are that methodology, applied and shipped.

I work in Rust when correctness matters (deterministic, zero-dependency, forbid(unsafe)), across the stack when it doesn't, and I'm most useful where AI throughput meets senior-engineer judgment — where someone has to make the machine's output trustworthy.
